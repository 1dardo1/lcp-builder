# Principios y decisiones clave

Notas de método acumuladas durante el proyecto. Relacionado: [[ADR-001 - Seleccion de stack tecnologico]], [[ADR-002 - Secuenciacion de desarrollo entre plataformas]], [[00 - Indice|Modelo de Dominio]].

## Sobre el modelo de dominio

- **El orden topológico no es negociable**: ningún tipo se define antes que sus dependencias. Las violaciones se resuelven reordenando físicamente el documento, no con workarounds ni referencias adelantadas.
- **Nunca corregir un enum o nombre de campo solo por la prosa de la Wiki**: verificar siempre contra JSON real antes de corregir. La prosa tiene errores de transcripción recurrentes (`IActiveEffects[]` → `IActiveEffectData[]`, `resistance` → `resist`, `Aux` → `Auxiliary`).
- **Validación de gramática ≠ evaluación de expresión**: evaluar una `DiceExpression` requiere datos de personaje en tiempo de ejecución, imposible en un contexto de solo creación de contenido. Se valida el patrón, se pospone la evaluación.
- Un solo ejemplo real no es suficiente para corregir un enum ya establecido (ver caso `damage.type` en minúscula en `pilot_gear.json`) — se anota como pendiente, no se aplica el cambio.

## Sobre arquitectura y capas

- El **dominio solo recibe rutas de archivo**, sin importar cómo se obtuvieron (selector nativo, drag-and-drop, cámara). Los fallos de adapter se comunican hacia arriba como errores genéricos, sin exponer la causa de plataforma.
- El diseño en Figma puede ir por delante del código Dart porque depende del modelo de dominio (documento), no del dominio compilado.
- La investigación de plugins de plataforma debe preceder a la implementación de adapters, para no invalidar trabajo si el dominio revela nuevas necesidades.
- Objetos de valor grandes con muchos campos de tipo lista omiten intencionadamente la igualdad estructural completa — decisión pendiente de revisar con `equatable` o `package:collection`.
- **Comentarios de código vs. vault**: el vault es la única fuente de verdad para lógica de negocio (reglas del dominio Lancer/COMP-CON) — el código no la duplica, solo referencia la sección (`// Ver vault MdD §X`). Los comentarios en código se reservan para decisiones técnicas de la implementación Dart (por qué esta estructura de tipos, correcciones verificadas contra JSON real que protegen contra una regresión futura tipo "esto no es un typo"). Motivo: evitar que ambas fuentes diverjan con el tiempo — solo el vault se actualiza cuando cambia una regla.
- **Campos polimórficos por catálogo externo (`IBonusData.val`)**: cuando la forma de un campo depende de un id de un catálogo externo (no un `type` fijo conocido de antemano), el patrón ya establecido en otros proyectos (JSON Schema `discriminator`+`mapping`, Zod discriminated unions) es un registro id → forma. En Dart se resuelve con el mismo mecanismo de enum-con-campo-asociado que ya usa el proyecto para `jsonValue` — ver [[Decisión - catálogo Bonus List y resolución de IBonusData.val]].
- **Motor de formulario genérico — esquema a mano, no codegen**: Dart/Flutter no tiene reflexión utilizable en runtime (`dart:mirrors` no funciona en AOT), así que "genérico" solo puede ser data-driven (una descripción declarativa por entidad, interpretada por un único motor). Se descartó `build_runner`/codegen para escribir esa descripción: mezclaría dos aprendizajes distintos (diseñar la abstracción vs. tooling de generación de código) y el objetivo de este ejercicio es lo primero. Revisar esta decisión solo si la fricción de mantener esquemas a mano se vuelve real, no antes.
- **Variance de funciones genéricas en Dart**: una clase genérica `C<T>` con un campo `String Function(T)` no se puede invocar de forma segura a través de una referencia sin argumento de tipo (`C` se trata como `C<dynamic>`, y `(T) -> String` no es subtipo de `(dynamic) -> String` — falla en tiempo de ejecución, no de compilación). Solución: un método de instancia dentro de la propia clase genérica (`String labelFor(dynamic v) => displayLabel(v as T)`), donde `T` sigue fijado por la construcción del objeto.
- **`CatalogFieldSpec` sirve también para uniones cerradas, no solo catálogos externos grandes**: el mismo mecanismo usado para `BonusId` (~90 entradas, externo) resuelve igual de bien uniones discriminadas pequeñas y cerradas como `IResistanceData` (resist/vulnerability/immunity) o `IOtherEffectData` (overshield/hp/repair/cover) — basta con un enum de UI local en vez de un catálogo del dominio. No hacía falta un `FieldSpec` nuevo para "unión cerrada de N vías"; era el mismo problema con menos entradas.
- **Completar `IWeaponData` en el formulario exigió dos piezas nuevas del motor, no solo repetir el patrón**: `MultiEnumFieldSpec` (selección de un subconjunto de un enum — `allowedTypes`, `damageTypes`...) y `GroupFieldSpec` (sub-formulario de forma fija, una sola instancia, no repetible — `IEffectSaveData`, o el lado "estructurado" de un `TextOrActiveEffect`). El comentario original que decía "esto repetiría el mismo patrón sin aportar mecanismo nuevo" (para `actions`/`activeEffects`/`synergies`/`deployables`/`profiles`) resultó parcialmente optimista — casi todo sí repetía el patrón, pero `TextOrActiveEffect` y los campos `num | NumericOrFormulaValue` de `IDeployableData` no encajaban en las piezas existentes.
- **Recursión acotada por decisión consciente, no por limitación técnica**: `IDeployableData.deployables` (un deployable puede contener otros deployables) se excluyó del formulario a 1 nivel de profundidad — el propio dominio ya anota que en la práctica "no crea jerarquías reales" (mismo criterio que Action/ActiveEffect). El motor sí podría soportar la recursión (un builder de esquema puede llamarse a sí mismo), pero se decidió no pagar ese coste de complejidad por un caso que no ocurre en contenido real — documentado explícitamente en el propio esquema, no omitido en silencio.
- **Reutilización de sub-esquema entre entidades hermanas**: `IWeaponProfile` duplica casi literalmente los campos de `IWeaponData` (mismo nombre, mismo tipo: `damage`, `range`, `actions`, `bonuses`, `synergies`, `deployables`, `onAttack`...). En vez de escribir el esquema dos veces, se extrajo una función compartida (`_weaponEffectFields()`) reutilizada tanto en el arma como en cada ítem de `profiles` — las keys son relativas al contexto de lectura/escritura activo, así que no colisionan entre el nivel superior y el ítem de lista.
- **`file_selector` frente a `file_picker` para el adapter de "selector nativo"**: el criterio decisivo no fue cobertura de plataformas (`file_picker` cubre más, incluido web) sino fiabilidad de la implementación en Linux/Android — `file_selector` es el paquete federado del propio equipo de Flutter y llama a APIs nativas del SO, mientras que `file_picker` en Linux ejecuta comandos de shell (`zenity`/`kdialog`/`qarma`, más frágil ante el entorno) y tiene un bug conocido donde `saveFile` no llega a escribir nada en Linux. Reafirma el principio ya escrito arriba ("el dominio solo recibe rutas de archivo"): el adapter vive en `presentation/`, no en `infrastructure/`, porque pedir la ruta es una interacción con el usuario, no una operación de E/S pura.
- **Generalizar "Crear" a 24 entidades sin tocar el motor**: el puerto `ContentPackExporter` pasó de un método específico (`exportWeapons`) a uno genérico (`export({manifest, content})` con `content: Map<String, List<Object>>`) — el puerto no necesita conocer las 24 entidades, solo su implementación en `infrastructure/` (que sí las conoce, con un `switch` de tipo que despacha al mapper JSON correspondiente). El caso de uso se generalizó igual (`CrearContenidoUseCase`, con `contentKey`/`content` en vez de un parámetro `weapon` tipado). La pantalla se resolvió con un patrón de configuración (`EntityCrearConfig`: título, `contentKey`, esquema, ensamblador, extractores de `id`/`name`) en vez de una pantalla por entidad — sin tipo común entre las 24 entidades del dominio, `Object` en el límite genérico es inevitable, pero se mantiene contenido a un puñado de puntos de despacho (`switch`/`as`), no esparcido por el código.
- **Trampa de closures en cadena con el mismo parámetro reescrito**: escribir `idOf: (content) => (content as X).id` y luego, en el campo siguiente, `nameOf: (content) => content.name` (asumiendo que `content` "ya" era `X` por el campo anterior) falla en compilación — cada closure tiene su propio parámetro `content`, tipado `Object` de forma independiente; no hay narrowing entre closures distintas aunque estén una al lado de la otra. Cada una necesita su propio cast.
- **Bug real encontrado por el cliente al probar en COMP/CON: el manifest identifica el content pack, no el archivo**: `generar_lcp_pruebas_entidades.dart` generaba 8 `.lcp` (uno por entidad) pero los 8 compartían el mismo manifest (mismo `name`/`author`/`version`) — COMP/CON los trata como el mismo pack, así que cargar los 8 en secuencia sobrescribía cada vez el anterior en vez de sumarlos (Content Pack Management mostraba solo 1 entrada de "entidades simples" en vez de 8, y el contenido activo era el del último cargado). Fix: manifest con `name` único por archivo (incluye el nombre de la entidad). Lección para cuando el cliente cargue varios `.lcp` propios a la vez: cada content pack necesita identidad de manifest propia, un archivo distinto no basta.
- **Bug relacionado, mismo hallazgo**: varios `id` de prueba coincidían por accidente con contenido oficial real de Lancer (`GMS` como manufacturer, `tg_accurate`, `st_shredded`) — un `.lcp` de prueba con esos ids se confunde con el contenido core en vez de aparecer como nuevo. Los datos de prueba deben usar ids que no puedan colisionar con catálogos reales (prefijo `TEST_`/`_test_`), no solo nombres que "suenen bien".
- **Extraer un módulo compartido solo cuando un segundo/tercer consumidor real lo confirma**: el "paquete" actions/bonuses/synergies/deployables/counters/activeEffects vivía dentro de `weapon_form_schema.dart` mientras solo lo usaba arma — no se extrajo antes por especulación ("podría hacer falta"), sino al llegar Reserve/CoreBonus/Talent, que repetían el mismo paquete literalmente. La extracción a `common_entity_fields.dart` se verificó primero con la suite de tests completa en verde (comportamiento idéntico de `weapon_form_schema.dart` ya reducido) antes de construir las 3 entidades nuevas encima — así un fallo posterior no podía deberse al propio refactor. Resultado: las 3 entidades se ensamblaron sin fricción de mecanismo nuevo, confirmando que la abstracción capturaba lo que de verdad se repetía (no una generalización prematura).
- **Caso 6 del catálogo (variabilidad por tier de NPC) resuelto sin `FieldSpec` nuevo**: los 4 tipos (`TierValue`, `NpcSize`, `EidolonShardCount`, `ICustomStatData.defaultValue`) parecían necesitar una pieza de motor propia, pero se descompusieron en dos piezas ya existentes — `ShapeChoiceFieldSpec` generalizado de A/B fijos a N ramas (cubre 2 formas en unos casos, 3 en `EidolonShardCount`), y `GroupFieldSpec` reutilizado tal cual para pedir "exactamente 3 campos, uno por tier" (mismo criterio ya usado para descartar `ListFieldSpec` en `IEffectSaveData`: conjunto fijo, no lista de tamaño libre). Se descartó una alternativa de inferir la forma a partir de qué casillas se rellenan (sin selector explícito) por introducir un estado ambiguo real (relleno parcial) que un selector explícito evita de raíz. La diferencia de tipo final entre ramas (`List<num>` vs. string `"X/Y/Z"`) se resuelve en el ensamblador de cada entidad, no en el esquema — ver [[Decisión - variabilidad por tier de NPC (caso 6 del catálogo de polimorfismo)]].
- **Generalizar `ShapeChoiceFieldSpec` a N ramas resultó servir también para el caso 1/2 (unión discriminada), no solo el 6**: al implementar el selector de variante de `IPilotGearData` (3 tipos: Weapon/Armor/Gear) e `INpcFeatureData` (5 tipos), la misma composición `ShapeChoiceFieldSpec` + `GroupFieldSpec` ya construida para `TierValue`/`EidolonShardCount` encajó sin cambios — el "discriminador por tag `type`" y la "variabilidad por tier" son problemas de dominio distintos, pero en el formulario ambos se reducen a "elige una rama de N, cada una con sus propios campos". Confirma que la pieza generalizada capturaba el patrón real (elegir entre formas alternativas), no un caso demasiado específico del tier de NPC.
- **Cierre de "Crear" — las 24 entidades completas sin ninguna pieza de motor genuinamente nueva desde `IWeaponData`**: `CatalogFieldSpec`, `ShapeChoiceFieldSpec` (ahora N ramas), `GroupFieldSpec` y `MultiEnumFieldSpec` —las 4 piezas ya construidas para completar el esquema de arma— resultaron suficientes para las 23 entidades restantes, incluidas las 8 con casos polimórficos propios. Dato a favor de la decisión original de invertir en un motor genérico data-driven en vez de una pantalla por entidad: el coste de las piezas del motor se amortizó sobre 24 entidades, no solo sobre la primera.
- **`IMechSystemData`/`IWeaponModData` — extraer un value bag (`MechSystemBaseValues`) cuando dos constructores de dominio distintos comparten todos los campos de uno**: `IWeaponModData` extiende (en el sentido de "repite todos los campos de", no herencia Dart de instancias) `IMechSystemData` más los suyos propios. Como son clases de dominio distintas, no se puede construir un `IMechSystemData` y "convertirlo" en `IWeaponModData` — cada ensamblador tiene que invocar su propio constructor con los ~17 campos base. Se extrajo `mechSystemBaseFields()`/`MechSystemBaseValues`/`mechSystemBaseFromValues()` a `common_entity_fields.dart` para no duplicar esa extracción de valores dos veces — mismo criterio de "extraer con un segundo consumidor real" ya aplicado al paquete de actions/bonuses.

## Sobre priorización y feedback

- El feedback del cliente es un **desempate**, no un primer filtro. Orden real: coste de arrastre → severidad de impacto → incertidumbre sobre ese coste de arrastre → coste de tiempo comparativo.
- Ante empate entre valor de producto y valor de producción, gana producción (dado que no hay plazos y el objetivo prioritario es aprendizaje) — salvo urgencia real y confirmada del cliente.
- Posponer indefinidamente algo que solo aporta valor de producción equivale, en la práctica, a descartarlo.

## Sobre el propio proceso de mentoría

- Modo socrático para decisiones de diseño que el equipo de desarrollo debe razonar; implementación directa una vez la decisión arquitectónica ya está tomada.
- Excepciones al modo socrático: redacción de documentos formales que recogen decisiones ya tomadas, tareas administrativas/de formato, o petición explícita y justificada.
- Variantes menores de nomenclatura se resuelven de forma autónoma cruzando JSON real; solo se pausa ante tipos genuinamente nuevos sin resolver o contradicciones semánticas reales.
