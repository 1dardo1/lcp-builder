## 15. NPC Data


> **Nota de alcance — contenido NO incluido por defecto en COMP/CON:** a diferencia de Pilot Data y Licensed Data, la fuente indica explícitamente que los datos NPC del Core Book requieren comprar el libro y descargar el "NPC LCP" por separado — no vienen precargados en la aplicación. Relevante para el ADR-003: si el caso de uso del cliente es generar contenido para su propio piloto/mech (no para que el máster cree enemigos), este bloque entero podría tener prioridad baja. Verificar con el cliente antes de asignarle fase.

### 15.1 NPC Classes and Templates

**Fuente:** NPC Classes and Templates

> **Correcciones de transcripción aplicadas sin pausa:** `active_effects?: IActiveEffects[]` → `IActiveEffectData[]`, en ambos tipos (`INpcClassData` e `INpcTemplateData`).
>
> **Nota de empaquetado, no de dominio — "collection" vs "library" style:** la fuente describe dos formas de organizar archivos NPC dentro del LCP (un archivo por clase/template con sus features incluidas, vs. archivos únicos `npc_classes.json`/`npc_features.json`/`npc_templates.json` con referencias por ID). Esto es un detalle de cómo se empaqueta el `.lcp` final, no una diferencia de tipo de dato del dominio — el modelo de dominio (`INpcClassData`, `INpcTemplateData`, `INpcFeatureData`) es el mismo independientemente del estilo de empaquetado elegido. Relevante para la fase de implementación del export, no para el modelo de dominio en sí.

### TierValue
**Tipo:** Value Object — patrón repetido en casi todos los campos de `stats` de `INpcClassData`: un valor puede ser un único entero (aplicado a los 3 tiers por igual) o un array de exactamente 3 enteros (uno por tier).

> Caso 6 de [[19. Catálogo de casos polimórficos (para el formulario)]] — variabilidad por tier, eje distinto de los casos 1-5 (no confundir al diseñar el formulario). `NpcSize` (más abajo) y `EidolonShardCount` (§15.3) son variantes del mismo eje.

```
TierValue = number | [number, number, number]
// Si es number, equivale a aplicar el mismo valor a los 3 tiers.
// Si es array, debe tener EXACTAMENTE 3 elementos (Tier 1, 2, 3).
```

### NpcSize
**Tipo:** Value Object — caso especial, distinto de `TierValue`: un NPC puede tener varios tamaños válidos simultáneamente dentro de un mismo tier (a diferencia del resto de stats, que son un único valor por tier).

```
NpcSize = [number[], number[], number[]]
// Array de EXACTAMENTE 3 sub-arrays (uno por tier), cada uno con uno o más
// valores de tamaño válidos para ese tier. Valores válidos: 0.5 | 1 | 2 | 3.
// Ejemplos válidos: [[1],[2],[3]]  o  [[0.5,1],[0.5,1],[0.5,1,2]]
// Valores > 3 no soportados (pero no deberían causar crash). Valores < 0.5
// o fraccionarios distintos de 0.5 pueden causar comportamiento indefinido.
```

### INpcClassData
**Tipo:** Entidad de catálogo — el GM selecciona/instancia NPCs a partir de clases existentes; el estado de un NPC concreto en partida es responsabilidad de COMP/CON, no de este modelo de LCP.

```
INpcClassData: {
  id: string,                              // requerido — único globalmente
  name: string,                             // requerido
  role: enum { artillery, controller, defender, striker, support, tank },  // requerido — COMP/CON no soporta multi-rol
  info: {                                   // requerido
    flavor: string,        // texto de Compendium
    tactics: string,       // nota de estrategia para el GM
    terse: string          // resumen ultra-corto para el GM
  },
  stats: {                                  // requerido
    armor: TierValue,
    hp: TierValue,
    evade: TierValue,
    edef: TierValue,
    heatcap: TierValue,
    speed: TierValue,
    sensor: TierValue,
    save: TierValue,
    hull: TierValue,
    agility: TierValue,
    systems: TierValue,
    engineering: TierValue,
    size: NpcSize,                          // forma especial — ver NpcSize
    activations: TierValue
  },
  optionalClassMin?: number,                // default 0
  optionalClassMax?: number,
  optionalClassPerTier?: number,            // ver nota de cálculo aditivo más abajo
  active_effects?: IActiveEffectData[]
}
```

### INpcTemplateData
**Tipo:** Entidad de catálogo — mismo patrón que `INpcClassData`.

```
INpcTemplateData: {
  id: string,                               // requerido — único globalmente
  template: true,                            // requerido — literal true, usado para validación de seguridad en formatos de colección
  name: string,                              // requerido
  description: string,                      // requerido, v-html
  forceTag?: enum { Mech, Ship, Vehicle, Biological, Squad, Other },  // bloquea el tag NPC, impide modificación por el usuario
  prohibitTemplates?: string[],             // IDs de templates cuya aplicación conjunta con este se prohíbe
  optionalMin?: number,                     // default 0 — NO restringido a features del template, aplica a cualquier feature válida del NPC
  optionalMax?: number,
  optionalPerTier?: number,                 // ver nota de cálculo aditivo
  optionalClassMin?: number,
  optionalClassMax?: number,
  optionalClassPerTier?: number,
  caveat?: string,                          // reglas especiales no trackeadas mecánicamente por COMP/CON
  active_effects?: IActiveEffectData[]
}
```

> **Nota — cálculo aditivo de `optionalPerTier`/`optionalClassPerTier`:** si `optionalMin`/`optionalMax` están presentes junto con `optionalPerTier`, el valor de `optionalPerTier` se **multiplica por el tier actual del NPC y se suma** a min/max — no los sustituye. Ejemplo de la fuente: con `optionalMin=3, optionalMax=4, optionalPerTier=1`, en Tier 2 el NPC selecciona entre 5 (3+1×2) y 6 (4+1×2) features. La misma lógica aditiva aplica a `optionalClassPerTier` sobre `optionalClassMin`/`optionalClassMax`.

### 15.2 NPC Features

**Fuente:** NPC Features

> **Correcciones de transcripción aplicadas sin pausa:** `tags?: ITagData[]` → `ITagInstance[]`; `active_effects?: IActiveEffects[]` → `IActiveEffectData[]`; `on_attack`/`on_hit`/`on_crit`/`on_miss` con tipo `ActiveEffect` (singular informal) → `IActiveEffectData` (mismo patrón ya resuelto en Mech Weapons, sección 13.3).
>
> **Campos de `IActionData` no usados por NPCs:** la fuente indica explícitamente que NPCs ignoran `cost`, `pilot`, `synergy_locations`, `tech_attack`, `log` al reutilizar `actions`. `synergy_locations`/`tech_attack`/`log` no existen siquiera en el `IActionData` ya definido en este documento (son residuos de v2, coherente con los cambios v2→v3 ya confirmados en la página Home) — se documenta la nota igualmente por si aparecen en LCPs legacy.
>
> **Campos de `IDeployableData` no usados por NPCs:** `deactivation`, `recall`, `redeploy`, `instances`, `cost`, `mech`, `synergies`, `counters` se ignoran. Adicionalmente, cualquier stat entero de un Deployable de NPC puede usar `TierValue` (entero o array de 3) en vez de un entero fijo — extensión específica de este contexto.

### INpcDamageData
**Tipo:** Value Object — variante restringida de `IDamageData` (sección 1): el campo `damage` debe ser un array de exactamente 3 enteros (uno por tier), nunca una expresión de dados.

```
INpcDamageData: IDamageData con la siguiente restricción:
  damage: [number, number, number]   // SIEMPRE 3 enteros — nunca DiceExpression/dice string
```

### INpcFeatureData
**Tipo:** Entidad de catálogo — union discriminado por `type`, con 5 variantes que comparten una base común.

> Caso 1 de [[19. Catálogo de casos polimórficos (para el formulario)]]. El campo `effect?: string | IActiveEffectData` de la base común es además caso 3.

```
// Base común a las 5 variantes
INpcFeatureBase: {
  id: string,                  // requerido — único globalmente
  name: string,                 // requerido
  origin?: string,               // id de NPC Class/Template de origen — ver nota
  base?: boolean,                // default false — si true, asignado automáticamente vía clase/template
  deprecated?: boolean,
  effect?: string | IActiveEffectData,
  hide_active?: boolean,
  build_feature?: boolean,       // sin efecto en combate; siempre oculto en Active Mode
  mod?: string,                  // id de otra INpcFeatureData a la que esta feature modifica — ver nota
  tags?: ITagInstance[],
  actions?: IActionData[],       // NPCs ignoran cost/pilot/synergy_locations/tech_attack/log
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[]  // NPCs ignoran varios campos — ver nota arriba
}

type INpcFeatureData =
  | INpcFeatureBase & { type: 'trait' }
  | INpcFeatureBase & { type: 'system' }
  | INpcFeatureBase & { type: 'reaction'; trigger?: string }
  | INpcFeatureBase & { type: 'tech'; attack_bonus?: TierValue; accuracy?: TierValue }
  | INpcFeatureBase & {
      type: 'weapon';
      weapon_type: string,              // formato "{Size} {Type}", ej. "Superheavy Rifle" — solo UI hoy, mantener convención por si se usa mecánicamente en el futuro
      damage: INpcDamageData[],
      range: IRangeData[],
      attacks: TierValue,
      attack_bonus?: TierValue,
      accuracy?: TierValue,
      on_attack?: IActiveEffectData,
      on_hit?: IActiveEffectData,
      on_crit?: IActiveEffectData,
      on_miss?: IActiveEffectData
    }

// default si se omite `type`: 'trait'
```

> **Nota — `origin`:** id de la NPC Class/Template de la que proviene esta feature. Solo es estrictamente necesario en formato Library cuando el origen NO incluye esta feature en su `base_features`/`optional_features` — si se omite en ese caso, el LCP carga correctamente pero la feature no queda vinculada a su origen. En formato Collection no es necesario.
>
> **Nota — `mod`:** mecanismo de v3 (mismo concepto que `IWeaponModData` pero aplicado a NPC Features) donde una feature modifica a otra existente — sus propiedades (`effect`, `bonuses`, `tags`, `deployables`) se **añaden** a los de la feature objetivo (referenciada por `id` en `mod`), no la sustituyen.
>
> **Nota — `system` vs `trait`:** estructuralmente idénticos; `system` existe solo para agrupar visualmente en la UI de COMP/CON, sin diferencia de campos ni de comportamiento mecánico.

### 15.3 Eidolons

**Fuente:** Eidolons

> **Nota de alcance — contenido condicionado a un suplemento:** mismo caso que `Bonds` (sección 11.7): la fuente indica explícitamente que la creación de Eidolons solo está disponible si el suplemento "No Room For a Wallflower" GM LCP está instalado y activado. Relevante para la priorización del ADR-003 — verificar con el cliente si este suplemento aplica a su mesa antes de asignarle prioridad de fase.
>
> **Corrección de transcripción aplicada sin pausa:** `active_effects?: IActiveEffects[]` → `IActiveEffectData[]`.

### EidolonShardCount
**Tipo:** Value Object — patrón nuevo, distinto de `TierValue`: además de entero único o array de 3, admite un string literal especial con significado dinámico.

```
EidolonShardCount = number | [number, number, number] | 'hostile_characters'
// number: equivale a [N, N, N] (mismo valor los 3 tiers)
// array: un valor por tier
// 'hostile_characters': el número de shards se genera dinámicamente según
// la cantidad de PCs hostiles presentes en el encuentro — no es un valor
// fijo de catálogo, se resuelve en tiempo de ejecución por COMP/CON.
```

### IEidolonShardData
**Tipo:** Value Object — no tiene `id` propio; vive anidado dentro de un `IEidolonLayerData`.

```
IEidolonShardData: {
  count: EidolonShardCount,        // requerido
  detail: string,                   // requerido, v-html
  features: INpcFeatureData[],     // requerido
  tier?: number                     // si se omite, usa el tier del layer padre
}
```

### IEidolonLayerData
**Tipo:** Entidad de catálogo — cada Eidolon se compone de varias layers; la primera ("Core Layer") se asigna automáticamente a cualquier Eidolon.

```
IEidolonLayerData: {
  id: string,                       // requerido — único globalmente
  name: string,                      // requerido
  appearance: string,                // requerido, v-html
  hints: string,                     // requerido, v-html — dirigido a jugadores
  rules: string,                     // requerido, v-html — admite sintaxis sensible a tier {X/Y/Z}, ver nota
  features?: INpcFeatureData[],
  shards?: IEidolonShardData,
  active_effects?: IActiveEffectData[]
}
```

> **Nota — sintaxis `{X/Y/Z}` en `rules`:** distinta de `TierValue` (que estructura un campo numérico completo) y distinta de los bonus strings de `DiceExpression`/`NumericOrFormulaValue` (que sustituyen un valor por una estadística del personaje). Aquí la interpolación ocurre **dentro de un bloque de texto en prosa**: el motor de render debe sustituir `{X/Y/Z}` por `X` si el contexto es tier 1, `Y` si es tier 2, `Z` si es tier 3. Es una tercera forma de variabilidad por tier, distinta de las otras dos ya vistas en este documento — no se debe confundir ni fusionar con `TierValue`.

### 15.4 Extra Features (Custom Stats)

**Fuente:** Custom Stats — resuelve la entrada "Extra Features (extra_features.json)" del checklist original; el archivo de datos real se llama `custom_stats.json`, distinto del nombre listado en el índice de la Wiki.

> **Marca explícita de la fuente — funcionalidad experimental:** la página empieza con la advertencia literal "WARNING: EXPERIMENTAL". Tratar este tipo con mayor cautela que el resto del documento — la propia spec admite que puede cambiar.
>
> **Limitación de alcance real de la plataforma (no de diseño propio):** la fuente confirma que COMP/CON no puede rastrear dependencias de stats entre LCPs distintos, por lo que cualquier stat nuevo debe quedar confinado a un único LCP — a diferencia de Tags, que sí se pueden compartir entre packs. Relevante si en el futuro (fase 4 del ADR-003, peticiones del cliente) se planteara definir un stat personalizado: la limitación no es una elección de este proyecto, es una restricción de la propia plataforma consumidora.
>
> **Cuarta forma de variabilidad por tier detectada en esta spec:** `default` puede ser un string con formato `"X/Y/Z"` (separado por barras), distinto de `TierValue` (array de 3 enteros, sin string), distinto del array de arrays de `NpcSize`, y distinto de la interpolación `{X/Y/Z}` dentro de prosa de Eidolons. Cuatro mecanismos distintos para expresar "varía por tier" coexisten en esta spec — cada uno se modela como su propio tipo en este documento, sin intentar unificarlos en una sola abstracción, ya que la propia fuente no lo hace.

### ICustomStatData
**Tipo:** Entidad — `key` identifica un stat persistente que se aplica a todas las entidades de combate que lo referencien; no es un catálogo de selección como `ITagData`, es una definición activa con efecto global dentro del LCP que la declara.

```
ICustomStatData: {
  key: string,                     // requerido — único globalmente, dentro del alcance del LCP (ver nota de limitación)
  title: string,                    // requerido — nombre de display
  trackable: boolean,               // requerido — true: stat current/max (como HP); false: stat solo-máximo (como Hull)
  default?: number | string,        // entero, o string "X/Y/Z" para valor por tier (NPC) — default 0 si se omite
  icon: string,                     // requerido — formato "mdi-ICON_ID" (MDI Icon set)
  sort: number                      // requerido — posición en UI de stat block; puede ser negativo para forzar primera posición
}
```
