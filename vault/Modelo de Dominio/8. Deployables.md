## 8. Deployables


**Fuente:** Deployables

> **Nota de nomenclatura verificada:** la página de `Deployables` enlaza a tipos llamados `IEffectStatusData`, `IEffectSpecialData`, `IEffectOtherData`, `IEffectResistData` para los campos `add_status`/`add_special`/`add_other`/`add_resist`. Se verificó entrando a las anclas referenciadas en la página de Active Effects: son los mismos tipos ya definidos en la sección 2 de este documento (`IStatusEffectData`, `ISpecialStatusData`, `IOtherEffectData`, `IResistanceData`), bajo un nombre de ancla distinto/desactualizado dentro de la propia Wiki. No se crean tipos nuevos ni alias — se reutilizan los ya existentes.
>
> **Corrección verificada contra ejemplos reales:** la prosa de esta página describe los defaults condicionales con `type = "drone"`/`"mine"` en minúscula, pero los tres ejemplos de código de la misma página usan mayúscula inicial (`"Drone"`, `"Mine"`, `"Deployable"`). Se adopta la grafía de los ejemplos reales, consistente con el resto del documento.

### DeployableType

A diferencia de otros enums de este documento, `type` admite **cualquier string libre** — pero dos valores concretos (`Drone`, `Mine`) activan reglas de negocio especiales (defaults distintos, manejo particular en Active Mode) documentadas más abajo. No es un enum cerrado.

```
DeployableType = string  // libre; "Drone" y "Mine" tienen tratamiento especial; default si se omite: "Deployable"
```

> Caso 5 de [[19. Catálogo de casos polimórficos (para el formulario)]] (string abierto con valores especiales reconocidos).

### IDeployableData
**Tipo:** Value Object — la propia fuente indica explícitamente que no tiene campo `id` (se genera uno único por instancia en tiempo de ejecución, no en la definición).

```
IDeployableData: {
  name: string,                       // requerido
  type?: DeployableType,              // default "Deployable"
  detail: string,                     // requerido
  activation?: ActivationType,        // default "Quick" — genera acción de Deploy en el item padre
  deactivation?: ActivationType,      // genera acción de Deactivation — elimina la instancia, SIN reembolsar cost
  recall?: ActivationType,            // genera acción de Recall — elimina la instancia, SÍ reembolsa cost
  redeploy?: ActivationType,          // genera acción de Redeploy, solo disponible tras recall — sin coste
  instances?: number,                 // default 1 — copias creadas por activación
  cost?: number,                      // default 1 si el item padre es "limited"
  size?: number,                      // default 0.5 si type="Drone" (ver tabla de defaults de hp más abajo)
  size_special?: string,              // string libre para descripción visual del tamaño (ej. "Line 4"), solo display
  // Campos number | NumericOrFormulaValue: caso 3 de
  // [[19. Catálogo de casos polimórficos (para el formulario)]] (forma decidida por el propio valor).
  armor?: number | NumericOrFormulaValue,   // default 0
  hp?: number | NumericOrFormulaValue,      // ver tabla de defaults condicionales
  evasion?: number | NumericOrFormulaValue, // default 10 si type="Drone", 5 si type="Mine"
  edef?: number | NumericOrFormulaValue,    // default 0, u 8 si type="Mine"
  heatcap?: number | NumericOrFormulaValue, // sin heatcap = todo el daño de heat se trata como daño Energy
  repcap?: number | NumericOrFormulaValue,
  sensor_range?: number | NumericOrFormulaValue,
  tech_attack?: number | NumericOrFormulaValue,
  save?: number | NumericOrFormulaValue,
  speed?: number | NumericOrFormulaValue,
  grapple?: number | NumericOrFormulaValue,
  attack_bonus?: number | NumericOrFormulaValue,
  damage?: IDamageData[],
  range?: IRangeData[],
  actions?: IActionData[],            // solo disponibles mientras el deployable está activo
  bonuses?: IBonusData[],             // solo activos mientras el deployable está activo
  synergies?: ISynergyData[],         // solo activas mientras el deployable está activo
  deployables?: IDeployableData[],    // sin anidación real: se inyectan en el tracker del PC/NPC propietario, no en este deployable
  counters?: ICounterData[],          // visibles solo en el propio deployable; se eliminan si el deployable se elimina/destruye
  active_effects?: IActiveEffectData[],
  add_status?: IStatusEffectData[],
  add_special?: ISpecialStatusData[],
  remove_special?: string[],
  add_other?: IOtherEffectData[],
  add_resist?: IResistanceData[],
  tags?: ITagInstance[],              // ver sección 6 (Tags) — ITagInstance, no ITagData: ITagData es el catálogo, esto es la instancia de uso
  pilot?: boolean,                    // default true si el item padre es Pilot Equipment, false en otro caso
  mech?: boolean                      // default false si el item padre es Pilot Equipment, true en otro caso
}
```

### Tabla de defaults condicionales según `type`

| Campo | Default general | Si `type = "Drone"` | Si `type = "Mine"` |
|---|---|---|---|
| `size` | — | 0.5 | — |
| `evasion` | — | 10 | 5 |
| `edef` | 0 | 0 | 8 |
| `hp` | según `size` (ver tabla siguiente) | 10 (fijo, no depende de `size`) | según `size` (ver tabla siguiente) |

### Tabla de defaults de `hp` según `size` (cuando `type` NO es `"Drone"`)

| `size` | `hp` por defecto |
|---|---|
| 0.5 | 10 |
| 1 | 20 |
| 2 | 30 |
| 3+ | 40 |

> **Nota de anidación:** `deployables` dentro de `IDeployableData` no crea jerarquías reales — la propia fuente aclara que un deployable emitido por otro deployable se inyecta en el tracker del PC/NPC propietario (no en el deployable padre), y que recall/redeploy desde un deployable anidado se redirige siempre al propietario original. Es el mismo patrón de "anidación que en la práctica es plana" que ya se documentó para `IActionData`/`IActiveEffectData`, pero aquí el límite lo impone explícitamente el motor de COMP/CON, no solo la ausencia de un campo.
