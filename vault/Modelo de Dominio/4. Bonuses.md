## 4. Bonuses


**Fuente:** Bonuses

> **Nota de verificación contra datos reales:** la página de la Wiki de `Bonuses` documenta valores para `weapon_types`, `weapon_sizes` y `range_types` que difieren de los enums "reales" ya definidos en este documento (vistos en Damage/Range/Weapons). Se verificó contra `lib/weapons.json` (datos reales del Core book) para distinguir errores de tecleo de valores genuinos de filtro:
> - `CQC` (visto en esta página) es un error de tecleo — el valor real es `CQB` (confirmado contra JSON).
> - `Aux` (anotado anteriormente en este documento para `WeaponSize`) era el error — el valor real es `Auxiliary` (confirmado contra JSON). **Corregido en la sección 1 (`WeaponSize`), donde el tipo se definió formalmente por necesidad anticipada.**
> - `any`, `Improvised` (weapon_types) y `Melee` (range_types) NO son errores: son valores de filtro válidos, propios del contexto de `Bonuses`, que no corresponden a ningún tipo real de arma (`Improvised` = armas de oportunidad sin entrada en `weapons.json`; `any`/`Melee` = comodines de filtro). No aparecen en `lib/weapons.json` porque no son tipos de item, son valores de consulta. Por esto se modelan como enums de filtro separados de los enums "reales", en vez de fusionarse en un único enum compartido.
> - Para `damage_types` y `range_types`, la fuente especifica que omitir el campo equivale a "todos" — **no** documenta un valor literal `any` dentro del array (a diferencia de `weapon_types`/`weapon_sizes`, donde `any` sí es un valor explícito admitido). Esta distinción se respeta tal cual la documenta la fuente.

### BonusWeaponTypeFilter (enum)

```
enum BonusWeaponTypeFilter { Rifle, Cannon, Launcher, CQB, Nexus, Melee, Improvised, any }
// any = comodín, aplica a todos los weapon_types. Default si se omite el campo.
```

### BonusWeaponSizeFilter (enum)

```
enum BonusWeaponSizeFilter { Auxiliary, Main, Heavy, Superheavy, any }
// any = comodín, aplica a todos los weapon_sizes. Default si se omite el campo.
```

### BonusRangeTypeFilter (enum)

```
enum BonusRangeTypeFilter { Threat, Range, Burst, Blast, Cone, Line, Melee }
// Sin valor "any" — la fuente no lo documenta para este campo.
// Si se omite el campo range_types completo, se interpreta como "todos".
```

### DieRoll
**Tipo:** Value Object — gramática distinta de `DiceExpression`: solo suma/resta de tiradas y enteros encadenados, sin multiplicación, división, paréntesis, ni bonus strings entre llaves. Usado únicamente en `IBonusData.val` cuando `id = "overcharge"`.

```
DieRoll = string  // forma: XdY+N, encadenable con + y - (ej. "1d6", "2d20-2", "1d6+2d20-2+3d6+1d20")
// X = número de dados, Y = caras del dado, N = bonus plano entero (opcional)
// A diferencia de DiceExpression: NO admite *, /, paréntesis, ni {bonus_strings}
```

### NumericOrFormulaValue
*(ya definido en la sección 2 — reutilizado aquí sin cambios)*

`IBonusData.val` usa este mismo tipo: número fijo, o string evaluado como expresión aritmética con bonus strings entre llaves (`{ll}`, `{grit}`, etc.). La lista de nombres válidos dentro de las llaves es específica de este contexto — ver tabla "Special Values" más abajo — pero no cambia la *forma* del tipo, solo qué valores son válidos dentro de él (igual que `DamageType` restringe qué strings son válidos en `IDamageData.type`).

**Tabla de Special Values admitidos en `val` (sustituyen al string completo entre llaves):**
`ll`, `tier`, `grit`, `size`, `hp`, `current_hp`, `speed`, `evasion`, `edef`, `tech_attack`, `overshield`, `armor`, `heatcap`, `heat`, `stress`, `current_stress`, `structure`, `current_structure`, `repcap`, `sensor`, `save`, `sp`, `hull`, `agi`, `sys`, `eng`.

### MountAssignment
**Tipo:** Value Object — extraído del formato de string `add_mount`, mismo criterio que `DiceExpression` (string con sub-componentes tipados, no un string libre).

> **Nota — no confundir con `MountType` (sección 13.2):** el enum de `mount_type` aquí es propio de este campo (`add_mount`, visto en `Bonuses`) y tiene un conjunto de valores ligeramente distinto al `MountType` formal definido más adelante en Frames (`Main, Heavy, 'Aux/Aux', Aux, 'Main/Aux', Flex, Integrated`) — por ejemplo, este no incluye `Flex` ni `Integrated`. Son dos enums independientes de la spec, no el mismo tipo reutilizado; cada uno documentado en su contexto de origen.

```
MountAssignment: {
  mount_type: enum { Aux, 'Aux/Aux', 'Aux/Main', Main, Heavy, Superheavy },
  max_mounts: number  // entero — umbral de mounts totales por debajo del cual se añade el mount adicional
}
// Formato de string original: "mount_type:max_mounts", ej. "main:3"
// NOTA: aquí "Aux" SÍ es el valor real documentado por la fuente para este
// campo concreto (add_mount), distinto del caso de WeaponSize. No se debe
// "corregir" a Auxiliary aquí sin verificar — son dos campos distintos
// de la spec, cada uno con su propia forma documentada.
// VERIFICADO contra lib/frames.json (campo `mounts`): Main, Flex, Heavy,
// Aux/Aux, Main/Aux confirmados contra datos reales. "Aux" suelto no
// apareció en la muestra revisada pero se acepta por simetría con Aux/Aux.
```

### IBonusData
**Tipo:** Value Object — `id` referencia el Bonus List (catálogo cerrado de IDs reconocidos por COMP/CON), no identidad de instancia.

```
IBonusData: {
  id: string,                              // debe ser uno de los IDs del Bonus List (ver tabla completa en la fuente original) o un id de Status/Condition
  val: NumericOrFormulaValue | boolean | DieRoll[] | MountAssignment,
  // El tipo real de `val` depende de qué ID se use (ver columna "Values" del Bonus List):
  //   - integer  -> NumericOrFormulaValue
  //   - boolean  -> boolean
  //   - DieRoll[] (solo "overcharge")
  //   - string especial (solo "add_mount") -> MountAssignment
  accuracy?: number,
  damage_types?: DamageType[],              // sin filtro "any" — omitir = todos
  range_types?: BonusRangeTypeFilter[],      // sin filtro "any" — omitir = todos
  weapon_types?: BonusWeaponTypeFilter[],    // default 'any' si se omite
  weapon_sizes?: BonusWeaponSizeFilter[],    // default 'any' si se omite
  overwrite?: boolean,
  replace?: boolean
}
// damage_types, range_types, weapon_types, weapon_sizes son EXCLUSIVOS entre
// sí: un item debe satisfacer TODAS las condiciones presentes para recibir
// el bonus (no es "cualquiera de ellas", es "todas las presentes").
```

> **PENDIENTE — DieRoll vs DiceExpression:** la página define un formato "DieRoll" (`XdY+N`, encadenable con `+`/`-`) muy similar a `DiceExpression`, usado solo en el bonus `overcharge` (tipo `DieRoll[]`). Es prácticamente idéntico a `DiceExpression` pero sin el componente de bonus-string entre llaves. Decisión pendiente de validar: ¿se modela como alias de `DiceExpression`, o se confirma como tipo distinto? No bloquea el resto de `IBonusData`, que no depende de este caso particular salvo para el ID `overcharge`.
