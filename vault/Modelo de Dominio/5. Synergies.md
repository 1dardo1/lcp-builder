## 5. Synergies


**Fuente:** Synergies

> **Nota de reutilización:** a diferencia de `Bonuses` (sección 4), esta página usa explícitamente los enums "reales" `WeaponType`/`WeaponSize` (sección 1) para sus filtros, no los filtros de `Bonuses`. El mecanismo de "omitir = todos" es el mismo que en `damage_types`/`range_types` de Bonuses: no hay un valor `any` dentro del array, simplemente se omite el campo.

### SystemType (enum)
**Fuente:** README inicial — sin página propia en la Wiki, documentado junto a `systems.json`.

```
enum SystemType { AI, Deployable, Drone, 'Flight System', Shield, System, Tech }
```

### SynergyLocation (enum especial)

Conjunto cerrado de ubicaciones donde puede aparecer un hint de synergy. La mayoría son valores fijos; `action_X` tiene una parte variable (`X` = string, ID de una acción existente) — distinto del patrón `round_start_X`/`round_end_X` de `EffectDuration`, donde `X` es un entero.

```
SynergyLocation =
  | 'rest' | 'weapon' | 'system' | 'deployable' | 'drone' | 'move' | 'boost'
  | 'structure' | 'armor' | 'hp' | 'overshield' | 'stress' | 'heat' | 'repair'
  | 'core_power' | 'overcharge' | 'hull' | 'agility' | 'systems' | 'engineering'
  | 'pilot_weapon' | 'cascade'
  | 'action_X'   // X = string, debe coincidir con el id de una Action existente (actions.json o equivalente)
```

> Caso 5 de [[19. Catálogo de casos polimórficos (para el formulario)]] (string con parte fija + parte variable).

### ISynergyData
**Tipo:** Value Object

```
ISynergyData: {
  locations: SynergyLocation[],     // requerido — al menos una ubicación
  detail: string,                   // requerido, v-html
  weapon_types?: WeaponType[],      // omitir = todos ("any")
  weapon_sizes?: WeaponSize[],      // omitir = todos ("any")
  system_types?: SystemType[]       // omitir = todos ("any")
}
```

> **Nota de uso (regla de la fuente, no inferencia propia):** las Synergies se documentan explícitamente como mecanismo de último recurso. La fuente indica usarlas solo para reglas contextuales que no se pueden representar con otros campos (Actions, Active Effects, Bonuses) — nunca para duplicar algo que ya vive en esos campos. Esta regla es relevante para cuando se diseñe el formulario de creación: no debería incentivarse su uso por encima de los campos estructurados ya existentes.
