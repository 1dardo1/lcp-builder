## 6. Tags


**Fuente:** Tags (definición del catálogo) + uso observado en `weapons.json`/`frames.json` (definición de la instancia, sin nombre propio documentado en la Wiki)

> **Distinción verificada contra múltiples fuentes:** el campo `tags?: ITagData[]` que aparece en `weapons`, `systems`, `pilot_gear`, etc. se documenta como un array de `ITagData` — pero `ITagData`, según la página `Tags` de la Wiki, es el objeto de **catálogo** (`{id, name, description, hidden?, filter_ignore?}`), no el objeto pequeño `{id, val}` que realmente aparece en los ejemplos de items reales (ej. `{"id": "tg_reliable", "val": 2}` en `weapons.json`). La Wiki nombra el catálogo, pero no nombra formalmente el objeto de instancia pequeño que en la práctica es el que rellena el campo `tags` de cada item. Se mantiene `ITagData` para el catálogo (nombre de la fuente) y se introduce `ITagInstance` (nombre propio, sin equivalente documentado) para el objeto pequeño.

### ITagInstance
**Tipo:** Value Object — `id` referencia una entrada del catálogo `ITagData` (de éste o de `tags.json` de un LCP cargado), no identidad de instancia propia. Mismo patrón ya visto con `IStatusEffectData`.

```
ITagInstance: {
  id: string,           // debe coincidir con el id de una entrada de ITagData (core data o LCP tags.json cargado)
  val?: string | number // sustituye el token {VAL} en la descripción de la entrada del catálogo, si la tiene
}
```

> **Nota — token `{VAL}`:** si la descripción de la entrada de catálogo (`ITagData.description`) contiene el token literal `{VAL}`, se sustituye por el valor de `val` al renderizar. El valor se inserta siempre como string (ej. `2 + 2` se muestra como "2 + 2", no como "4").

### ITagData
**Tipo:** Entidad de catálogo — `id` identifica una entrada única y persistente del catálogo de tags (core data o `tags.json` de un LCP). No es una instancia de uso; es la definición compartida que las instancias (`ITagInstance`) referencian. Esto la distingue de otros catálogos ya vistos como value object: aquí no hablamos de "la instancia que aparece en un arma", hablamos del propio catálogo, que sí tiene una vida propia editable (un LCP puede añadir, y en teoría versionar, sus propias entradas de tag).

```
ITagData: {
  id: string,                // requerido — identificador único en el catálogo
  name: string,              // requerido
  description: string,       // requerido, v-html — puede contener el token {VAL}
  hidden?: boolean,           // usado para features de UI interna; no se puede añadir desde una LCP, pero SÍ se puede asignar a contenido custom (ver tabla de hidden tags conocidos)
  filter_ignore?: boolean     // evita que este tag aparezca en los filtros de equipo de la UI
}
```

**Hidden tags conocidos (asignables a contenido custom aunque `hidden` no se pueda crear libremente):**

| Tag ID | Comportamiento |
|---|---|
| `tg_set_max_uses` | Establece el máximo de usos del item |
| `tg_set_damage_type` | Establece el tipo de daño del item |
| `tg_set_damage_value` | Establece el valor de daño del item |
