## 13. Licensed Data


### 13.1 Manufacturers

**Fuente:** Manufacturers

> **Hallazgo de transcripción:** el ejemplo real (`HA` / Harrison Armory) incluye un campo `logo` no documentado en la definición formal de esta página — probablemente residuo de v2 visto también en el README inicial muy al principio de esta investigación. No se incluye en `IManufacturerData` por no estar en la definición formal v3. El mismo ejemplo tampoco usa `icon_svg`/`icon_url`, coherente con que ambos son opcionales.

### IManufacturerData
**Tipo:** Entidad de catálogo — referenciado por `source` desde Core Bonuses, Frames, y todo el equipo licenciado (Weapons, Systems, Mods). `id` debe ser único globalmente y es la clave por la que se referencia desde todo el resto del LCP.

```
IManufacturerData: {
  id: string,          // requerido — forma acrónimo/abreviada (ej. GMS), referenciado en campo `source` de items licenciados
  name: string,         // requerido
  description: string,  // requerido, v-html
  quote: string,         // requerido, v-html
  light: string,         // requerido — código hex (#XXXXXX), para temas de UI con base 'light'
  dark: string,          // requerido — código hex (#XXXXXX), para temas de UI con base 'dark'
  icon_svg?: string,     // SVG inline, sanitizado — preferido sobre icon_url
  icon_url?: string      // fallback si no hay icon_svg, también sanitizado
}
```

> **Nota — excepciones de `source`:** la propia fuente indica que tres categorías de equipo NO requieren `source`/manufacturer asociado: Special Equipment, Integrated Equipment, y Exotic Equipment. Estas tres categorías corresponden a la entrada "Integrated, Special and Exotic Equipment" del checklist — pendiente de definir más adelante en esta misma sección.

### 13.2 Frames

**Fuente:** Frames + Frame Traits + Core Systems (ambos dependencias genuinas, no variaciones de nombre)

> **Correcciones de transcripción aplicadas sin pausa:**
> - `IFrameTraitData.active_effects?: IActiveEffects[]` (plural) → `IActiveEffectData[]`, mismo error recurrente.
> - `ICoreSystemData.active_effects?: IActiveEffects[]` (plural) → `IActiveEffectData[]`, mismo error.
> - `ICoreSystemData.tags?: ITagData[]` → `ITagInstance[]`, mismo error de catálogo-vs-instancia ya resuelto en sección 6.
> - `ICoreSystemData.passive_bonuses?: IActionData[]` — error genuino de la fuente (tipo equivocado, no nombre): por contexto y por la descripción de la propia página ("An array of Bonuses..."), debe ser `IBonusData[]`. Corregido.
> - `ICoreSystemData.passive_synergies` aparece sin `?` en la definición formal pero se describe como opcional en prosa — se trata como opcional por coherencia.

### MountType (enum)

Tipo formal, ya usado de pasada (sección 4, `MountAssignment`) pero nunca definido como entidad propia hasta ahora — esta página da la lista completa y autoritativa.

```
enum MountType { Main, Heavy, 'Aux/Aux', Aux, 'Main/Aux', Flex, Integrated }
```

### IFrameTraitData
**Tipo:** Value Object — agrupa el contenido mecánico de un trait; no tiene `id` propio.

```
IFrameTraitData: {
  name: string,                          // requerido
  description: string,                  // requerido, v-html
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],                 // no ocupa espacio de mount, no se puede quitar
  special_equipment?: string[],          // no se instala automáticamente, debe instalarse manualmente
  active_effects?: IActiveEffectData[]
}
```

### ICoreSystemData
**Tipo:** Value Object — cada Frame tiene exactamente uno; no tiene `id` propio.

```
ICoreSystemData: {
  name: string,                          // requerido
  description?: string,                  // opcional pero recomendado, v-html
  active_name: string,                   // requerido
  active_effect: string,                 // requerido, v-html
  activation: ActivationType,            // requerido
  deactivation?: ActivationType,         // si se omite, permanece activo hasta `use` o fin de misión
  use?: enum { Round, 'Next Round', Scene, Encounter, Mission },  // si se omite, activo hasta deactivation o fin de misión
  active_effects?: IActiveEffectData[],  // añadidos al mech al activar el Core Power
  active_actions?: IActionData[],
  active_bonuses?: IBonusData[],
  active_synergies?: ISynergyData[],
  passive_name?: string,
  passive_effect?: string,               // v-html
  passive_actions?: IActionData[],       // siempre disponibles
  passive_bonuses?: IBonusData[],        // siempre activos — corregido, ver nota de transcripción
  passive_synergies?: ISynergyData[],    // siempre activas
  deployables?: IDeployableData[],       // usables desde el panel del Core System
  counters?: ICounterData[],             // siempre presentes
  integrated?: string[],                 // siempre instalado, independiente del estado de activación
  special_equipment?: string[],          // siempre instalado, independiente del estado de activación
  tags?: ITagInstance[]                  // específicos del Core System
}
```

### IPrerequisite
**Tipo:** Value Object — usado solo dentro de `specialty` de `IFrameData`.

```
IPrerequisite: {
  source: string,        // requerido — Manufacturer ID
  min_rank: number,      // requerido — rango mínimo de licencia de ese manufacturer que el piloto debe tener desbloqueado
  cumulative?: boolean   // default false — ver nota
}
```

> **Nota — `cumulative`:** si es `true`, el piloto debe acumular al menos `min_rank` **sumando** licencias de ese manufacturer (patrón similar al de gating de CORE Bonuses). Si es `false` (default), basta con tener una sola licencia de ese manufacturer al rango `min_rank`.

### IFrameData
**Tipo:** Entidad de catálogo — el piloto/cliente selecciona y desbloquea licencias existentes, no crea instancias de frame con estado propio (el estado de un mech concreto en partida es responsabilidad de COMP/CON, no de este modelo de LCP).

```
IFrameData: {
  id: string,                              // requerido — único globalmente
  name: string,                             // requerido
  source: string,                           // requerido — debe coincidir con un Manufacturer ID
  license_id?: string,                      // requerido SOLO SI variant está presente — debe coincidir con el id del frame principal de la licencia
  license_level: number,                    // requerido — 0 a 3
  mechtype: string[],                       // requerido — al menos uno; solo hint de UI, sin efecto mecánico
  description: string,                      // requerido, v-html
  mounts: MountType[],                      // requerido — al menos uno
  stats: {
    size: number,           // requerido — entero positivo o exactamente 0.5 (único valor fraccionario válido)
    structure: number,      // requerido — entero positivo
    stress: number,         // requerido — entero positivo
    armor: number,          // requerido — entero positivo
    hp: number,              // requerido — entero positivo
    evasion: number,        // requerido — entero positivo
    edef: number,            // requerido — entero positivo
    heatcap: number,        // requerido — entero positivo
    repcap: number,         // requerido — entero positivo
    sensor_range: number,   // requerido — entero positivo
    tech_attack: number,    // requerido — entero positivo
    save: number,            // requerido — entero positivo
    speed: number,           // requerido — entero positivo
    sp: number               // requerido — entero positivo
  },
  traits: IFrameTraitData[],                // requerido — puede ser array vacío
  core_system: ICoreSystemData,             // requerido — exactamente uno
  specialty?: boolean | IPrerequisite,      // union discriminado por forma, no por campo — ver nota
  variant?: string,                          // id (preferido) o name (legacy) del frame del cual este es variante
  image_url?: string,
  y_pos?: number                            // alineación vertical en banners de UI
}
```

> **Nota — `specialty` como union sin discriminador de campo:** a diferencia de los otros unions discriminados de este documento (`IResistanceData` por exclusión mutua de campos, `IOtherEffectData` por el campo `type`), aquí el discriminador es la **forma del valor en sí**: si es `boolean`, oculta el frame asociado y marca la licencia como "Specialty"; si es un objeto `IPrerequisite`, define una condición de gating. No hay campo `type` que lo indique — se distingue en tiempo de ejecución comprobando si el valor es booleano o un objeto.
>
> **Nota — regla invertida sobre `license_id` respecto a publicaciones anteriores del README:** esta página establece que `license_id` es requerido cuando `variant` está presente (para enlazar la variante a su licencia principal) — no al revés. Verificar este matiz si se encuentran fuentes adicionales que lo contradigan.

### 13.3 Mech Equipment — Mech Weapons (weapons.json)

**Fuente:** Weapons (página ya leída íntegramente en esta sesión — ver ejemplo `mw_smartgun`)

> **Resolución de `EffectObject`:** la página de Weapons usa `string | EffectObject` para `effect`/`on_attack`/`on_hit`/`on_crit`/`on_miss`, enlazando a una sección genérica titulada "Effect Objects" — que en todas las páginas de este documento resulta ser el mismo enlace a la página `Active Effects`, ya definida como `IActiveEffectData` (sección 2). No existe un tipo `EffectObject` formalmente distinto y separado — es el nombre informal de sección usado para referirse al mismo tipo. Se usa `IActiveEffectData`.
>
> **Correcciones de transcripción aplicadas sin pausa:** `tags?: ITagData[]` → `ITagInstance[]` (mismo error recurrente). `WeaponSize`/`WeaponType` ya corregidos en sección 1.

### IWeaponProfile
**Tipo:** Value Object — todos los campos salvo `name` son opcionales; si se omite un campo, hereda el valor del arma contenedora. No se puede anidar (un profile no puede contener otros profiles).

```
IWeaponProfile: {
  name: string,                              // requerido — único campo realmente obligatorio
  effect?: string | IActiveEffectData,
  skirmish?: boolean,
  barrage?: boolean,
  cost?: number,
  on_attack?: string | IActiveEffectData,
  on_hit?: string | IActiveEffectData,
  on_crit?: string | IActiveEffectData,
  on_miss?: string | IActiveEffectData,
  active_effects?: IActiveEffectData[],
  damage?: IDamageData[],
  range?: IRangeData[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],
  special_equipment?: string[]
}
```

> **Nota de uso:** para un arma con varios profiles, el profile por defecto es el de índice 0 del array (usado para ordenar por daño, etc.). Actions/Effects/Synergies de un profile solo están disponibles mientras ese profile esté activo.

### IWeaponData
**Tipo:** Entidad de catálogo — el piloto selecciona/equipa armas existentes de licencias desbloqueadas, no crea instancias con estado propio (más allá de lo que gestione COMP/CON en runtime, fuera del alcance de este LCP).

```
IWeaponData: {
  id: string,                                // requerido — único globalmente
  name: string,                               // requerido
  source: string,                             // requerido — debe coincidir con un Manufacturer ID (salvo excepciones, ver nota de Manufacturers)
  license: string,                            // requerido — nombre de display de la licencia contenedora
  license_id: string,                         // requerido — debe coincidir con el id de un Frame
  license_level: number,                      // requerido — 0 a 3
  effect: string,                             // requerido, v-html — texto de reglas
  description: string,                        // requerido, v-html
  mount: MountType,                           // requerido — un único valor, no array (a diferencia de Frames.mounts)
  type: WeaponType | WeaponType[],            // requerido
  ammo?: IAmmoData[],
  cost?: number,                               // requerido SOLO si el item tiene tag "limited" — default 1
  barrage?: boolean,                          // default true salvo Superheavy
  skirmish?: boolean,                         // default true salvo Superheavy
  no_attack?: boolean,                        // suprime generación de acciones Skirmish/Barrage
  no_mods?: boolean,                          // impide añadir mods a este arma
  no_core_bonus?: boolean,                    // suprime bonificadores de mount-sensitive core bonuses
  damage?: IDamageData[],
  range?: IRangeData[],
  tags?: ITagInstance[],
  sp?: number,                                 // entero, default 0
  on_attack?: string | IActiveEffectData,
  on_hit?: string | IActiveEffectData,
  on_crit?: string | IActiveEffectData,
  on_miss?: string | IActiveEffectData,
  active_effects?: IActiveEffectData[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  no_bonus?: boolean,                         // default false — inhibe recolección/display de Bonuses
  synergies?: ISynergyData[],
  no_synergy?: boolean,                       // default false — inhibe recolección/display de Synergies
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],
  special_equipment?: string[],
  profiles?: IWeaponProfile[]                 // máximo 1 nivel de anidación — ver IWeaponProfile
}
```

> **Nota — `mount` singular vs. `mounts` plural de Frames:** a diferencia de `IFrameData.mounts` (array, el frame admite varios tipos de mount), `IWeaponData.mount` es un único valor — el arma ocupa exactamente un tipo de mount concreto, no varios alternativos.
>
> **Nota — defaults de `barrage`/`skirmish` invertidos respecto a una omisión simultánea:** si ninguno de los dos se especifica, `barrage` es `true` para todas las armas, y `skirmish` es `true` para todas excepto las de tamaño Superheavy. Si solo uno de los dos se especifica explícitamente, no se infiere automáticamente el valor del otro — ambos deben evaluarse de forma independiente contra sus propios defaults.

### 13.4 Mech Equipment — Mech Systems (systems.json)

**Fuente:** Systems

> **Correcciones de transcripción aplicadas sin pausa:** `tags?: ITagData[]` → `ITagInstance[]`; `active_effects?: IActiveEffects[]` → `IActiveEffectData[]`. Mismos errores recurrentes.
>
> **Matiz nuevo — excepción de `source`/`license`/`license_id` por License Collection:** distinto de la excepción ya vista en Manufacturers (Special/Integrated/Exotic Equipment, que nunca requieren `source`). Aquí la excepción es contextual: si el item se distribuye dentro de un "License Collection" (formato aún no definido en este documento — ver Pendientes), estos tres campos se vuelven opcionales porque el contexto del collection ya los implica.
>
> **Riesgo conocido reafirmado:** la fuente repite explícitamente la advertencia ya vista en Core Bonuses sobre `integrated`: COMP/CON no detecta referencias circulares y la app puede fallar (crash) si se produce una.

### IMechSystemData
**Tipo:** Entidad de catálogo — mismo patrón que `IWeaponData`/`IFrameData`.

```
IMechSystemData: {
  id: string,                          // requerido — único globalmente
  name: string,                         // requerido
  source?: string,                      // requerido SALVO si el item viene en un License Collection — debe coincidir con un Manufacturer ID
  license?: string,                     // requerido SALVO License Collection — nombre de display de la licencia contenedora
  license_id?: string,                  // requerido SALVO License Collection — debe coincidir con el id de un Frame
  license_level: number,                // requerido — 0 a 3
  type?: SystemType,                    // default 'System' si se omite
  effect?: string,                      // v-html
  description?: string,                 // v-html
  sp?: number,                           // entero, default 0
  tags?: ITagInstance[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  no_bonus?: boolean,                   // default false
  synergies?: ISynergyData[],
  no_synergy?: boolean,                 // default false
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],                // ver nota de riesgo — sin validación de referencias circulares
  special_equipment?: string[],
  active_effects?: IActiveEffectData[]  // se aplican al instalar el system, independientes de su estado de activación
}
```

### 13.5 Mech Equipment — Mech Weapon Mods (mods.json)

**Fuente:** Weapon Mods

> **Corrección de nomenclatura aplicada sin pausa:** la fuente usa `ISystemData` como tipo base de extensión — mismo error de transcripción de nombre ya visto en otras páginas; el tipo real ya definido en este documento es `IMechSystemData` (sección 13.4). Se usa ese nombre.
>
> **`added_tags` usa `ITagData[]` en la fuente** — mismo error recurrente, corregido a `ITagInstance[]`.
>
> **Regla de composición no trivial — `added_damage`/`added_range`:** si el tipo de daño/rango del mod coincide con uno ya presente en el arma base, los dados se **suman** al daño/rango existente (ej. arma con `2d6 Kinetic` + mod con `1d6 Kinetic` = `3d6 Kinetic`). Si el tipo no coincide, el mod se **añade como entrada separada** (ej. arma con `2d6 Kinetic` + mod con `1d6 Explosive` = `2d6 Kinetic` + `1d6 Explosive` por separado). Esta regla de fusión/append condicional por tipo es lógica de negocio real, no solo una concatenación simple de arrays — debe implementarse explícitamente, no asumirse.
>
> **`restricted_types`/`restricted_sizes` deprecados aquí** (a diferencia de `Ammo`, sección 9, donde son vigentes — ver nota cruzada en esa sección). La fuente indica que COMP/CON los transforma internamente a `allowed_*`; se incluyen en el tipo solo por compatibilidad con LCPs v2, no se recomienda su uso en contenido nuevo.

### IWeaponModData
**Tipo:** Entidad de catálogo — extiende todos los campos de `IMechSystemData` (sección 13.4), añadiendo los campos específicos de mod listados abajo.

```
IWeaponModData: IMechSystemData & {
  on_miss?: IActiveEffectData,
  on_attack?: IActiveEffectData,
  on_hit?: IActiveEffectData,
  on_crit?: IActiveEffectData,
  allowed_types?: WeaponType[],          // omitir = permitido en todos los tipos
  allowed_sizes?: WeaponSize[],          // omitir = permitido en todos los tamaños
  added_tags?: ITagInstance[],           // se eliminan si el mod se quita o destruye
  added_damage?: IDamageData[],          // ver nota de regla de composición — se elimina si el mod se quita o destruye
  added_range?: IRangeData[],            // ver nota de regla de composición — se elimina si el mod se quita o destruye
  restricted_types?: WeaponType[],       // DEPRECADO — tiene precedencia sobre allowed_sizes (no allowed_types)
  restricted_sizes?: WeaponSize[]        // DEPRECADO — tiene precedencia sobre restricted_types
}
```

> **Nota — precedencia cruzada entre `restricted_types` y `restricted_sizes`:** la fuente indica que `restricted_types` "overrides restricted_sizes", y a su vez `restricted_sizes` "overrides restricted_types" — una afirmación circular tal como está escrita en la página. Se transcribe literalmente tal como aparece en la fuente; dado que ambos campos están deprecados, se recomienda no depender de esta regla de precedencia en contenido nuevo en vez de intentar resolver la aparente contradicción.

### 13.6 Integrated, Special and Exotic Equipment

**Fuente:** página redirige a Home — no existe definición de tipo propia separada.

> **Resolución — sin tipo nuevo:** estas tres categorías no son estructuras de datos distintas. Son **mecanismos de instalación** ya vistos y usados repetidamente en este documento: los campos `integrated?: string[]` y `special_equipment?: string[]` (presentes en `ICoreBonusData`, `IFrameTraitData`, `ICoreSystemData`, `IMechSystemData`, `IWeaponModData`...) son arrays de IDs que referencian equipo de Mech Equipment **ya existente** (`IWeaponData`, `IMechSystemData`, `IWeaponModData`). No hay un `IIntegratedEquipmentData` o similar — "integrado" y "especial" describen el **modo de instalación** de un item ya tipado en otra parte (automático e irremovible vs. disponible para instalación manual), no un tipo de item nuevo.
>
> **Exotic Equipment — confirmado como sinónimo de Special Equipment (verificado contra FAQ y README oficiales de COMP/CON):** "Exotic" NO es un tipo de dato propio ni una tercera categoría a nivel de modelo. La lista del piloto se llama literalmente "special/exotic equipment" — una sola lista para ambos. El mecanismo de datos es idéntico al de Special Equipment (`special_equipment?: string[]`, array de IDs). La única diferencia es de **presentación de UI**: COMP/CON oculta los Exotics tras una opción avanzada ("Show Exotic items in the Compendium") por el sistema de honor, ya que se desbloquean durante la partida. A efectos del modelo de dominio y del formulario de creación, Exotic = Special Equipment; la distinción "exotic" se expresaría, si acaso, mediante un tag, no mediante un campo o tipo distinto. **Pendiente cerrado.**
>
> **Confirmaciones útiles obtenidas de la página Home, aplicables a correcciones ya hechas en este documento:** la fuente confirma explícitamente que los campos `on_miss`/`on_attack`/`on_hit`/`on_crit` de Weapons fueron reemplazados por Effect Objects en v3, con conversión automática de strings legacy — esto confirma la resolución de `EffectObject` = `IActiveEffectData` ya aplicada en la sección 13.3. También confirma textualmente que `tables.json` se renombró a `lists.json` — resuelto sin tipo nuevo en el checklist de la sección "Other" (ver más abajo).
