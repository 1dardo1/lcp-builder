## 17. Other


### 17.1 Manifest

**Fuente:** Manifest

> **Inconsistencia de nombre de campo en la propia fuente:** el bloque de definición formal llama al campo `version_history`, pero la sección de descripción narrativa lo llama `changelog` en su título de sección, aunque el cuerpo de esa misma sección sigue refiriéndose a `version_history` como el nombre del array. Se usa `version_history`, el nombre del bloque de definición — más fiable que un título de sección, por el mismo criterio ya aplicado en otros casos de esta investigación (estructura formal > prosa).

### SemverConstraint
**Tipo:** Value Object — string con gramática propia, mismo criterio que `DiceExpression`/`MountAssignment`. No es un semver libre: admite dos modificadores especiales con semántica propia.

> Caso 5 de [[19. Catálogo de casos polimórficos (para el formulario)]].

```
SemverConstraint = string
// Formas válidas:
//   "X.Y.Z"   -> versión mínima aceptada; COMP/CON acepta esa versión O CUALQUIER POSTERIOR
//   "*"       -> acepta cualquier versión, siempre que el nombre coincida
//   "=X.Y.Z"  -> acepta EXCLUSIVAMENTE esa versión exacta, ninguna otra
// Ejemplo: si el usuario tiene instalada la v2.0.0 y la dependencia
// declarada es "1.0.1", COMP/CON la acepta (2.0.0 >= 1.0.1).
// Si la dependencia declarada fuera "=1.0.1", NO aceptaría la 2.0.0.
```

### IChangelogItem
**Tipo:** Value Object

```
IChangelogItem: {
  version: string,    // requerido — semver estricto X.Y.Z, sin modificadores
  date: string,        // requerido — formato YYYY-MM-DD
  changes: string[]    // requerido — se renderiza como lista no ordenada en la app
}
```

### ILcpDependency
**Tipo:** Value Object

```
ILcpDependency: {
  name: string,              // requerido — debe coincidir EXACTAMENTE con el campo `name` del manifest de la dependencia
  version: SemverConstraint,  // requerido
  link?: string               // recomendado aunque opcional — ayuda al usuario a encontrar/descargar la dependencia
}
```

> **Nota — orden y validación automática de dependencias:** COMP/CON ordena y carga automáticamente los LCPs según sus dependencias declaradas, y no carga un LCP si alguna dependencia declarada no está satisfecha — el usuario es notificado en el "Content Pack Manager". Esto es responsabilidad de COMP/CON, no algo que el LCP deba implementar — pero el `name` de cada dependencia debe ser exacto para que el mecanismo funcione.

### ILcpManifestData
**Tipo:** Entidad — un único manifest por LCP, identificado de facto por la combinación `name` + `version` (no tiene un campo `id` explícito, a diferencia del resto de catálogos de este documento). Obligatorio en todo LCP — sin él, COMP/CON no puede cargar el contenido.

```
ILcpManifestData: {
  name: string,                          // requerido
  author: string,                        // requerido
  description: string,                  // requerido, v-html
  version: string,                       // requerido — semver estricto X.Y.Z
  image_url?: string,                    // típicamente la portada del libro o diseño similar
  website?: string,                      // requerido SOLO para envíos al Community Content Directory — debe ser una página de itch.io
  dependencies?: ILcpDependency[],
  v3?: boolean,                          // default false (equivale a "solo compatible con v2") — fuertemente recomendado true en LCPs nuevos
  version_history?: IChangelogItem[]
}
```

### 17.3 SITREPs

**Fuente:** Sitreps

> **Resuelto — campos de victoria confirmados contra múltiples fuentes:** la página de la Wiki consultada mostraba una definición incompleta (solo `id`, `name`, `description`, `deployment?`, `objective?`, `extraction?`, `conditions?`). Se verificó contra múltiples fuentes independientes de la spec (npm `@massif/lancer-data`, Skypack, Openbase, fork whitespine): `pcVictory`, `enemyVictory` y `noVictory` son parte **oficial** del tipo, todos opcionales y v-html. No eran un campo "del ejemplo que faltaba en la definición" — la definición de la Wiki simplemente estaba desactualizada. Se incluyen como opcionales confirmados.
>
> **Nota:** el campo `conditions?: ISitrepCondition[]` aparecía en la versión de la Wiki pero NO en las definiciones de npm/Skypack/etc. Se mantiene por precaución (puede ser un añadido v3 posterior a esas publicaciones de npm, que son más antiguas), pero marcado como no confirmado de forma cruzada — inverso al caso de los campos de victoria.

### 17.4 Statuses & Conditions

**Fuente:** Statuses & Conditions

> **Cierre de referencia pendiente:** este es el tipo de catálogo al que apunta `id` en `IStatusEffectData` (sección 2) — ahí se documentó la referencia ("debe existir en core data o LCP statuses.json cargado") sin definir la entidad de catálogo en sí, porque dependía de esta página. Se cierra ahora ese círculo.
>
> **Corrección verificada contra el código fuente real (no solo la Wiki):** el "hallazgo de transcripción" original decía que `icon` era "mismo patrón legacy ya visto en Manufacturers (`logo`)" y se ignoraba — al revisar `src/classes/Status.ts` en `massif-press/compcon` (código actual), resulta que `icon` sí es un campo real y requerido (id corto usado para construir `cc:${type}_${icon}`, el icono de catálogo). Mismo patrón que se confirmó en Manufacturers (13.1): el campo del ejemplo real no era residuo, era la propia definición actual sin documentar aquí. Además, el campo de SVG personalizado se llama `svg`, no `icon_svg` — y `IStatusData` no tiene ningún equivalente a `icon_url` (no soporta icono personalizado por URL, solo `icon` de catálogo o `svg` inline).

### IStatusConditionData
**Tipo:** Entidad de catálogo.

```
IStatusConditionData: {
  id: string,                            // requerido — único globalmente; es el id referenciado por IStatusEffectData.id (sección 2)
  name: string,                           // requerido
  type: enum { Status, Condition },      // requerido — case-sensitive
  effects: string,                        // requerido, v-html
  terse?: string,                         // preferir el menor número de caracteres posible
  icon: string,                           // requerido en la interfaz TS real — id corto de icono de catálogo (cc:${type}_${icon}). No modelado todavía en este proyecto.
  svg?: string,                           // SVG inline, sanitizado. Antes documentado (mal) como "icon_svg".
  exclusive?: enum { Mech, Pilot }        // si está presente, restringe a quién puede aplicarse — case-sensitive
}
```

> **Pendiente de decidir:** este proyecto ya tiene un campo `iconUrl`/`icon_url` en `IStatusConditionData` (Dart) que no corresponde a nada real en COMP/CON — no rompe nada (COMP/CON simplemente lo ignora, no lo lee), pero tampoco hace nada. Se deja tal cual por ahora (no es el bug reportado), pendiente de decidir si se quita o se sustituye por el `icon` de catálogo real.

### 17.5 Lists

**Fuente:** Lists — confirma la fusión ya anotada: `tables.json` (v2) se renombró a `lists.json` (v3), mismo contenido.

### IListsData
**Tipo:** Entidad — único objeto por LCP (no es un array de instancias como el resto de catálogos); cada clave es un array de strings rolleable, ampliando las opciones ya presentes en COMP/CON sin sustituirlas.

```
IListsData: {
  pilot_names?: string[],    // rolleable en la primera pantalla de Create New Pilot
  callsigns?: string[],      // rolleable en la primera pantalla de Create New Pilot
  team_names?: string[],     // rolleable en el diálogo Add New Group
  mech_names?: string[],     // rolleable en la pantalla Add New Mech to Hangar
  quirks?: string[]          // tabla rolleable de Quirks para pilotos Flash Cloned
}
```
