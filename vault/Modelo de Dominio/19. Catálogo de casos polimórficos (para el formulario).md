## 19. Catálogo de casos polimórficos (para el formulario)

> **Nota de naturaleza distinta al resto del documento:** las secciones 1-18 transcriben la spec. Esta sección no añade tipos nuevos — reagrupa tipos ya definidos bajo un criterio distinto (cómo se decide su forma concreta), pensado para diseñar el formulario de Crear y su motor de renderizado. Surge de la pregunta: "¿es viable un único modelo de formulario genérico para las 24 entidades?" — la respuesta corta es que sí, salvo por estos casos, que necesitan trato especial.

### ¿Qué es un "caso polimórfico" aquí?

Un campo o tipo cuya forma concreta (qué datos contiene, cómo se construye) no es fija — varía según algo. Lo relevante para el formulario no es *que* varíe, sino *cómo se entera el código de cuál de las formas posibles le toca*. Eso determina qué widget hace falta: un selector de opciones, un campo de texto con patrón, o algo que hoy no se puede resolver todavía.

---

### 1. Discriminador explícito por `type` (conjunto cerrado y conocido de antemano)

La propia spec incluye un campo `type` cuyo valor, de una lista fija, dice qué forma tiene el resto. En Dart, cada opción es una clase distinta dentro de una `sealed class`.

| Tipo | Discriminador | Variantes |
|---|---|---|
| `IPilotGearData` (§11.3) | `type` | `'Weapon'` → `IPilotWeaponData`, `'Armor'` → `IPilotArmorData`, `'Gear'` → `IPilotGearItemData` |
| `INpcFeatureData` (§15.2) | `type` (default `'trait'` si se omite) | `trait`, `system`, `reaction`, `tech`, `weapon` |
| `IOtherEffectData` (§2) | `type` | `overshield`, `hp`, `repair`, `cover` |

**Para el formulario:** el más sencillo de los cuatro casos. Un selector (desplegable o radio) con las opciones fijas: al elegir una, el formulario ya sabe exactamente qué campos pedir a continuación.

---

### 2. Discriminador implícito por qué campo está presente (exclusión mutua)

No hay un campo `type` — la forma se decide viendo cuál de varios campos hermanos tiene valor (solo uno debe estar presente).

| Tipo | Campos exclusivos |
|---|---|
| `IResistanceData` (§2) | exactamente uno de `resist` / `vulnerability` / `immunity` |

**Para el formulario:** en la práctica, igual que el caso 1 — un selector de opciones. La diferencia solo importa al construir el JSON final: no se escribe un campo `type`, se rellena el campo correspondiente y los otros dos se omiten.

---

### 3. Discriminador por la forma del propio valor (sin tag ni campo exclusivo)

Un único campo puede contener valores de naturaleza distinta (un número suelto vs. un objeto pequeño, un string vs. un booleano, un elemento vs. una lista), y no hay ninguna pista externa — se distingue mirando el valor en sí en tiempo de ejecución (en Dart: `is num`, `is bool`, `is List`...).

| Tipo / campo | Formas posibles |
|---|---|
| `NumericOrFormulaValue` (usado en `IDeployableData`: `armor`, `hp`, `evasion`, `edef`, `heatcap`, `repcap`, `sensorRange`, `techAttack`, `save`, `speed`, `grapple`, `attackBonus` — §8) | número fijo, o string-fórmula con bonus strings (`{grit}`, `{ll}`...) |
| `IFrameData.specialty` (§13.2) | `bool`, o un `IPrerequisite` completo |
| `IDamageData.save` / `INpcDamageData.save` (§1, §15.2) | string suelto, o `{ stat, aoe? }` |
| `ITagInstance.val` (§6) | string, o número |
| `IWeaponData.type` (§13.3) | un `WeaponType`, o una lista de ellos |
| `StringOrBool` (usado en varios `aoe`) | string, o booleano |

**Para el formulario:** hace falta un mini-selector previo ("¿número fijo o fórmula?", "¿sí/no o algo más concreto?") antes de mostrar el campo real — no hay una lista de nombres de "tipo" como en el caso 1, solo dos formas posibles y una decisión binaria.

---

### 4. Discriminador por catálogo externo (el caso difícil — depende de datos que no tenemos modelados todavía)

La forma depende de consultar una tabla externa por un id, y esa tabla no forma parte de este documento todavía.

| Tipo / campo | Depende de |
|---|---|
| `IBonusData.val` (§4) | qué `id` se use — el Bonus List de COMP/CON (catálogo cerrado con decenas de entradas, cada una con su tipo asociado: entero, booleano, `DieRoll[]`, o el especial `MountAssignment` para `"add_mount"`) |

**Para el formulario:** no se puede resolver solo mirando el valor en tiempo de ejecución, como en el caso 3 — hace falta primero traer/transcribir el Bonus List completo (pendiente, ver nota en §4 del vault). Sin eso, el formulario no puede saber qué campo mostrar después de que el usuario elija un id de bonus.

---

### 5. "Enums especiales" — string con parte fija + parte variable

Distinto de los casos 1-4: aquí el campo **siempre** es un string (nunca cambia de tipo de dato), solo cambia el contenido dentro de un patrón fijo. No hace falta un selector de "forma", sino validar un patrón (y a veces construir el string a partir de sub-campos más simples).

| Tipo | Patrón |
|---|---|
| `EffectDuration` (§2) | fijos + `round_start_X` / `round_end_X` (X = entero) |
| `ReactionFrequency` (§2) | `Unlimited` + `X/round`, `X/scene`, `X/mission` (X = entero) |
| `SynergyLocation` (§5) | fijos + `action_X` (X = id de una Action) |
| `DeployableType` (§8) | libre, con `Drone`/`Mine` reconocidos especialmente |
| `DiceExpression`, `DieRoll`, `MountAssignment`, `SemverConstraint` | gramáticas propias, ver secciones 1, 4, 4, 17 respectivamente |

**Para el formulario:** un campo de texto con validación de patrón, o (mejor UX) sub-campos separados que el formulario combina en el string final — ej. un selector "inicio/fin de ronda" + un número, en vez de pedir `"round_start_3"` en crudo.

---

### 6. Variabilidad por tier (NPC) — un eje distinto, no confundir con los anteriores

La spec de NPC (§15) ya anota que existen **cuatro mecanismos distintos** para expresar "esto varía según el tier 1/2/3 del NPC": `TierValue` (número o array de 3), `NpcSize` (array de arrays), `EidolonShardCount` (sesión de tier + el string especial `'hostile_characters'`), y `ICustomStatData.default` (string `"X/Y/Z"`). Comparten el rasgo superficial de "un campo con más de una forma posible", pero la razón de fondo es otra (tier del NPC, no discriminación de contenido) — se listan aquí solo para que no se confundan con los casos 1-5 al diseñar el formulario genérico.

---

### Resumen — cuándo se resuelve cada caso

| Caso | Se resuelve en... |
|---|---|
| 1. Tag `type` | Tiempo de compilación (Dart ya modela cada variante como clase distinta — `sealed class`) |
| 2. Campo exclusivo | Tiempo de compilación (igual que el 1, pero sin campo `type` explícito) |
| 3. Forma del valor | Tiempo de ejecución, mirando el propio dato (`is num`, `is bool`...) |
| 4. Catálogo externo | No resuelto todavía — depende de datos pendientes de transcribir (Bonus List) |
| 5. Enum especial | Tiempo de ejecución, validando un patrón de string |
| 6. Tier NPC | Tiempo de ejecución, pero por un eje distinto (no es discriminación de contenido) |

**Pendiente de decisión (no resuelto en esta nota):** cómo tratará el motor de formulario genérico cada una de estas categorías — si las cubre todas de forma genérica, si excluye alguna (rompiendo el "un solo modelo para todo") y le da un widget especial, o algo intermedio. Se revisa y decide en conversación aparte.
