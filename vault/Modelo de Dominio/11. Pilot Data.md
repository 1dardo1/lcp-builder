## 11. Pilot Data


### 11.1 Backgrounds

**Fuente:** Backgrounds

### IBackgroundData
**Tipo:** Entidad de catálogo — mismo patrón que `ITagData`: el jugador elige una entrada existente de un catálogo fijo, no crea una instancia con estado propio. Se distingue de `ICounterData` (entidad con estado mutable persistente) en que aquí no hay un valor que cambie con el tiempo — la entidad es la propia definición, elegible pero no editable en juego.

```
IBackgroundData: {
  id: string,          // requerido — único globalmente
  name: string,        // requerido
  description: string, // requerido, v-html
  skills?: string[]    // opcional — array de IDs referenciando skills.json, por convención (no por tipo)
}
```

> **Nota de uso:** un background es puramente descriptivo (texto + referencias a skill triggers sugeridos). No tiene campos mecánicos propios — toda su función es de catálogo de selección en la creación del piloto.

### 11.2 Core Bonuses

**Fuente:** Core Bonuses

> **Corrección de nomenclatura aplicada sin pausa (criterio ya validado):** la firma usa `active_effects?: IActiveEffects[]` en plural — inconsistente con el patrón singular de nombrado en el resto del documento (`IActionData`, `IBonusData`, `ISynergyData`...) y con el propio tipo `IActiveEffectData` ya definido en la sección 2. Por contenido idéntico (mismos campos: `name`, `detail`, `condition`, `damage`, `add_status`...) se confirma que es el mismo tipo con un nombre de transcripción distinto en esta página. Se usa `IActiveEffectData[]`.

### ICoreBonusData
**Tipo:** Entidad de catálogo — mismo patrón que `IBackgroundData`/`ITagData`: el piloto elige una entrada de un catálogo fijo al subir de nivel, no crea una instancia con estado propio.

```
ICoreBonusData: {
  id: string,                         // requerido — único globalmente
  name: string,                       // requerido
  source: string,                     // requerido — debe coincidir con un Manufacturer ID
  effect: string,                     // requerido, v-html
  description: string,                // requerido, v-html
  mounted_effect?: string,            // texto para el panel de mount al instalar un Core Bonus relacionado con mount
  active_effects?: IActiveEffectData[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],              // IDs de Mech Equipment — instalado automáticamente, no removible salvo quitando el item padre
  special_equipment?: string[]        // IDs de Mech Equipment — disponible en el selector, NO instalado automáticamente
}
```

> **Nota de riesgo conocida (de la propia fuente):** COMP/CON no valida referencias circulares en `integrated`. Una referencia recursiva (un item integrado que a su vez integra al original, directa o indirectamente) no se detecta y provoca un fallo de la aplicación. Esto es responsabilidad del equipo de desarrollo evitar al diseñar el formulario de creación, no algo que la spec garantice.
>
> **Nota de uso:** según la propia fuente, muchos efectos de Core Bonus están hardcodeados en COMP/CON. Si los campos disponibles no pueden satisfacer mecánicamente el efecto deseado, debe quedar explícito en `effect`/`mounted_effect` (texto de reglas, no automatización).

### 11.3 Pilot Gear

**Fuente:** Pilot Gear

> **Correcciones de nomenclatura aplicadas sin pausa (criterio ya validado):** esta página repite los mismos dos errores de nomenclatura ya vistos en otras páginas — `tags?: ITagData[]` (debería ser `ITagInstance[]`, ver sección 6) y `active_effects?: IActiveEffects[]` (debería ser `IActiveEffectData[]`, ver sección 3). Se corrigen ambos sin pausa.
>
> **Hallazgo pendiente de verificar — `damage.type` en minúscula:** el ejemplo real `pg_archaic_melee` usa `"type": "kinetic"` (minúscula) dentro de su campo `damage`, mientras que `DamageType` (sección 1) tiene sus valores en mayúscula inicial (`Kinetic`). A diferencia de `CQB`/`Auxiliary` (donde verificamos contra múltiples ejemplos antes de corregir el enum), aquí solo hay un ejemplo visto hasta ahora. **No se corrige `DamageType` con un solo dato — se anota como pendiente de contrastar con más ejemplos reales antes de decidir si es un error puntual de este ítem concreto o si COMP/CON acepta minúsculas de forma case-insensitive.**

Pilot Gear son tres tipos diferenciados por `type` (mismo patrón de union discriminado ya usado en `IOtherEffectData`), agrupados en un único archivo `pilot_gear.json`.

### IPilotGearData
**Tipo:** Entidad de catálogo — mismo patrón que el resto de catálogos de equipo (`ITagData`, `IBackgroundData`, `ICoreBonusData`): el piloto selecciona/equipa una entrada existente del catálogo, no crea una instancia con estado propio en el sentido de `ICounterData`.

```
type IPilotGearData =
  | IPilotWeaponData
  | IPilotArmorData
  | IPilotGearItemData

IPilotWeaponData: {
  id: string,                          // requerido
  name: string,                        // requerido
  type: 'Weapon',                      // requerido, case-sensitive
  description?: string,
  effect?: string,
  range?: IRangeData[],
  damage?: IDamageData[],
  tags?: ITagInstance[],
  actions?: IActionData[],             // disponibles solo cuando el piloto está UNMOUNTED
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],     // disponibles solo cuando el piloto está UNMOUNTED
  counters?: ICounterData[],
  active_effects?: IActiveEffectData[] // se aplican al piloto, no al mech
}

IPilotArmorData: {
  id: string,                          // requerido
  name: string,                        // requerido
  type: 'Armor',                       // requerido, case-sensitive
  description?: string,
  tags?: ITagInstance[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[]
}

IPilotGearItemData: {
  id: string,                          // requerido
  name: string,                        // requerido
  type: 'Gear',                        // requerido, case-sensitive
  description?: string,
  tags?: ITagInstance[],
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[]
}
```

> **Nota — disponibilidad por estado de montura:** las `actions` y `deployables` de un `IPilotWeaponData` solo están disponibles cuando el piloto está **desmontado** (UNMOUNTED, es decir, fuera de su mech). Esta restricción no se aplica explícitamente a `IPilotArmorData`/`IPilotGearItemData` en esta página — pendiente de confirmar si aplica igual o es exclusiva del tipo Weapon.

### 11.4 Reserves

**Fuente:** Reserves

> **Corrección de nomenclatura aplicada sin pausa:** `active_effects?: IActiveEffects[]` (plural) corregido a `IActiveEffectData[]`, mismo error de transcripción ya visto en otras páginas.

### IReserveData
**Tipo:** Entidad de catálogo — mismo patrón que el resto de catálogos de equipo ya vistos.

```
IReserveData: {
  id: string,                          // requerido
  name: string,                        // requerido
  type: enum { Mech, Tactical, Resource, Bonus },  // requerido — solo agrupa en pestañas de UI, sin efecto mecánico
  label?: string,                       // default: el propio `name`
  description?: string,                // v-html
  consumable?: boolean,                // default false
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],
  special_equipment?: string[],
  active_effects?: IActiveEffectData[]
}
```

> **Nota de uso:** mientras la reserva esté equipada, sus elementos asociados (actions/bonuses/synergies/deployables/counters) están disponibles hasta que el jugador la marque manualmente como "Used" o la elimine. Si `consumable` es `true`, se desactiva automáticamente al usar cualquiera de las actions o deployables asociadas en Active Mode.

### 11.6 Talents

**Fuente:** Talents + Talent Rank (dependencia genuina, no variación de nombre — primera vez en este bloque que aparece un tipo realmente nuevo, no ya definido bajo otro nombre)

> **Corrección de nomenclatura aplicada sin pausa:** `active_effects?: IActiveEffects[]` (plural) en `IRankData` corregido a `IActiveEffectData[]`, mismo error recurrente.

### IRankData
**Tipo:** Value Object — agrupa el contenido mecánico de un rank concreto; no tiene `id` propio (el talento entero tiene `id`, no cada rank).

```
IRankData: {
  name: string,                          // requerido
  description: string,                  // requerido, v-html
  exclusive?: boolean,                   // default false — ver nota
  actions?: IActionData[],
  bonuses?: IBonusData[],
  synergies?: ISynergyData[],
  deployables?: IDeployableData[],
  counters?: ICounterData[],
  integrated?: string[],
  special_equipment?: string[],
  active_effects?: IActiveEffectData[]
}
```

> **Nota de comportamiento — `exclusive`:** este campo no describe el rank en sí; describe cómo se combina con los **otros ranks del mismo talento** al subir de nivel. Si `exclusive = true`, solo el equipo (`bonuses`/`deployables`/`counters`/`integrated`) del rank más alto desbloqueado está activo — los ranks anteriores quedan "sustituidos". Si `false` (default), el equipo de **todos** los ranks desbloqueados está activo simultáneamente. Es una propiedad relacional entre los tres ranks de un talento, no del rank aislado — al diseñar el formulario de creación, esto implica que la decisión de "exclusive sí/no" se toma una vez para todo el talento, aunque el campo viva en cada rank individualmente.

### ITalentData
**Tipo:** Entidad de catálogo — mismo patrón que el resto de catálogos elegibles por el piloto.

```
ITalentData: {
  id: string,           // requerido — único globalmente
  name: string,          // requerido
  description: string,  // requerido, v-html
  ranks: IRankData[],    // requerido — actualmente solo se soportan exactamente 3 ranks
  icon_svg?: string,     // SVG inline, sanitizado (DOMPurify) — preferido sobre icon_url
  icon_url?: string,     // fallback si no hay icon_svg, también sanitizado
  terse?: string         // descripción corta para UI condensada
}
```

> **Restricción de la fuente:** actualmente COMP/CON solo soporta talentos con exactamente 3 ranks. La página advierte que un número distinto "debería" funcionar pero no está probado — riesgo a tener en cuenta si el formulario de creación permite un número variable de ranks.

### 11.7 Bonds

**Fuente:** Bonds

> **Nota de alcance — contenido condicionado a un suplemento:** la propia fuente indica que la UI de Bonds "will not be shown unless the Karrakin Trade Baronies LCP is installed and activated" — es decir, este bloque no es parte del Core Book base, depende de que el usuario final tenga instalado un suplemento concreto (Karrakin Trade Baronies). Esto es relevante para la priorización del ADR-003: a diferencia del resto de Pilot Data (Backgrounds, Talents, etc., todos parte del Core Book), `Bonds` solo aporta valor si el cliente real usa ese suplemento — dato a verificar con él antes de asignarle prioridad de fase.

### IQuestionData
**Tipo:** Value Object

```
IQuestionData: {
  question: string,    // requerido, v-html
  options: string[]    // requerido — respuestas sugeridas, v-html
}
```

### IBondPowerData
**Tipo:** Value Object — incluso cuando se referencia desde `bond_powers.json` (ver nota de `origin` más abajo), el poder en sí no tiene `id` propio, solo el Bond contenedor lo tiene.

```
IBondPowerData: {
  name: string,                 // requerido
  description: string,          // requerido, v-html
  frequency?: ActionFrequency,  // ya definido en sección 2
  prerequisite?: string,        // v-html
  veteran?: boolean,            // default false — solo disponible en estatus Veteran
  master?: boolean,             // default false — solo disponible en estatus Master
  origin?: string                // ver nota — solo se usa en bond_powers.json
}
```

> **Nota — integridad referencial de `origin`:** este campo solo se usa cuando un `IBondPowerData` se define en un archivo separado `bond_powers.json` (en vez de anidado dentro de `IBondData.powers`). En ese caso, `origin` debe coincidir **exactamente** con el `id` de un `IBondData` existente. La fuente advierte explícitamente que poderes sin un `origin` válido no serán seleccionables por el usuario, y que poderes "universales" sin Bond asociado no están planeados como funcionalidad. Esta es una regla de validación cruzada entre dos archivos distintos del LCP, no solo dentro de un mismo objeto.

### IBondData
**Tipo:** Entidad de catálogo — mismo patrón que el resto de catálogos elegibles por el piloto.

```
IBondData: {
  id: string,                  // requerido — único globalmente
  name: string,                 // requerido
  major_ideals: string[],       // requerido — típicamente 2-5 elementos
  minor_ideals: string[],       // requerido — típicamente 2-5 elementos
  questions: IQuestionData[],   // requerido
  powers: IBondPowerData[]      // requerido
}
```
